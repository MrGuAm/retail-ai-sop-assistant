## 项目简介

本项目是一个基于零售商品管理场景的 AI 知识库 / ERP 操作 SOP 助手，主要用于模拟企业内部员工在处理商品资料、库存、调拨、销售数据等业务问题时，通过 AI 问答方式快速查询操作规范和排查思路。

项目不使用任何真实企业数据，所有内容均基于通用零售业务流程整理，适合作为 AI 应用实施、知识库搭建、业务流程梳理和自动化工具落地的练习项目。

## 项目背景

在零售企业中，商品管理相关工作通常涉及 ERP 系统、商品资料、SKU、门店库存、区域调拨、销售数据核对等流程。新人在刚接触业务系统时，容易遇到以下问题：

- 不清楚 ERP 操作前需要确认哪些信息；
- 不知道库存异常应该从哪些维度排查；
- 对调拨单、销售数据、商品资料之间的关系理解不清；
- 遇到数据不一致时，缺少标准化排查步骤；
- 业务经验依赖人工口头传递，难以沉淀复用。

因此，本项目尝试将常见业务流程整理为 SOP 文档，并通过 AI 知识库问答的方式，让用户可以用自然语言查询处理流程和注意事项。

## 真实问答示例

下面是一条完整示例问答，展示“回答演示”区域会如何输出真实内容：

```text
用户提问：
调拨单数量和实际到货数量不一致怎么办？

问题判断：
这类问题通常属于调拨单据流未闭环，或者调出、调入双方在商品编码、数量、单位、单据状态上存在不一致。

处理步骤：
1. 先核对调拨单号、商品编码、规格、单位和出入库数量是否一致。
2. 查看调拨申请、调拨出库、调拨入库三个环节是否都已完成审核。
3. 确认是否发生过少发、少收、补发、退回或重复操作。
4. 如存在实物差异，先留存照片和交接记录，再决定补发、退回或改单。

需要确认的信息：
- 调出方和调入方名称
- 调拨单当前状态
- 出库时间、入库时间、交接记录
- 是否存在在途库存未清

风险提醒：
- 未查清原因前直接调账，会掩盖真实问题。
- 只看出库不看入库，容易误判调拨已完成。
- 收货方不核对直接入库，会把错货固化为系统正确数据。

建议后续动作：
- 处理完成后复核双方库存是否同步恢复。
- 检查该单据是否仍挂在在途状态。
- 把异常原因沉淀到 FAQ 或培训材料里，减少重复问题。
```

## Dify / Coze 搭建建议

1. 创建一个新的知识库问答应用或 Bot。
2. 上传 `docs/` 下的 6 个 Markdown 文档。
3. 将 `prompts/assistant_prompt.md` 或 `07_知识库系统提示词模板.md` 作为系统提示词基础。
4. 设置回复必须固定输出 5 个模块。
5. 使用 [08_知识库测试问题集.md](./08_知识库测试问题集.md) 做上线前验收。

## 推荐回答风格

- 优先给出执行顺序，而不是泛泛解释
- 先提醒需要确认哪些信息，再给处理动作
- 对库存、金额、批量修改、删除、作废等高风险动作，明确提醒先确认权限或负责人
- 信息不足时，不硬答，先要求补充门店、SKU、时间范围、单据状态等背景
